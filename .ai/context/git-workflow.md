# Git Workflow

## Repository

- **GitHub**: https://github.com/betagouv/france-chaleur-urbaine
- **Primary branches**: `main` (production), `dev` (integration, default branch).

## Branch strategy

- `main` — production. Merging `main` is ideally a **fast-forward** from `dev`.
- `dev` — staging/integration (default branch, origin/HEAD). Minor fixes can be committed directly.
- Feature branches — branch from `dev`, merge via PR into `dev`.

Branch naming: descriptive names are fine (e.g., `improve_autocomplete_adresse`, `migration_node_24`). No strict prefix convention.

**Flow:**
1. Create feature branch from `dev`.
2. Open PR → Scalingo spawns a review app (reuses dev DB, no migrations).
3. CI runs (lint, types, tests, build).
4. Merge PR into `dev` → auto-deploy to staging.
5. When ready for production: fast-forward merge `dev` into `main` → auto-deploy to production.

## Commit conventions

No strict format (no Conventional Commits). Keep messages clear and focus on the **why**, not the what. The diff already shows what changed — the commit message should explain the reasoning.

Never include session URLs, "Co-authored-by" trailers, or any AI attribution in commit messages.

## Pre-commit checklist

Always run before committing:
```bash
pnpm lint      # Biome lint
pnpm ts        # TypeScript type check
pnpm test      # Run relevant tests
```

Secret detection: `.gitguardian.yaml` configured for pre-commit hooks.

## CI checks (GitHub Actions)

CI is configured in `.github/workflows/ci.yml`.

On every push :
1. Install dependencies (frozen lockfile)
2. Audit (continue on error)
3. Biome lint (`biome ci`)
4. Database migrations (test DB)
5. Vitest (unit + integration tests)
6. Build verification
7. Autodeploy to Scalingo

## Protected files

- **Never commit**: `.env.local`, credentials, API keys.
- **Review carefully**: `src/server/db/migrations/`, `package.json` changes, CI config.
- **Auto-generated (don't edit)**: `pnpm-lock.yaml`, `src/server/db/kysely/database.ts` (regenerated by `pnpm db:sync`).
